## Bug Summary

The blocking overlay stops firing after ~24-48 hours of an active block session. `MyAccessibilityService` continues running but `BlockingService` either loses its listener reference or gets killed by Android, resulting in silent failure with no overlay shown.

## Evidence

| Log/Screenshot | Status |
|----------------|--------|
| Overlay fires on session start | OK |
| Overlay fires after 1h | OK |
| Overlay fires after 24-48h | FAIL |
| Sentry error on service death | FAIL (silent failure — no error logged) |

---

## Reproduction

### Steps to Reproduce
1. Start a block session with at least one blocked app
2. Verify overlay fires when opening the blocked app
3. Leave the device idle (or in normal use) for 24-48 hours
4. Open the blocked app again

### Actual Behavior
The blocked app opens normally without any overlay. No Sentry error is logged. The block session still appears active in the JS layer.

### Expected Behavior
The blocking overlay should fire every time a blocked app is opened, regardless of how long the session has been active.

### Scenarios (Given/When/Then)

#### Bug Scenario
```gherkin
Given a block session has been active for 24-48 hours
  And "Instagram" is in the blocked apps list
When the user opens Instagram
Then no blocking overlay is displayed
  But the block session still shows as active in the app
```

#### Fixed Scenario
```gherkin
Given a block session has been active for 48+ hours
  And "Instagram" is in the blocked apps list
When the user opens Instagram
Then the blocking overlay is displayed within 500ms
  And BlockingService logs a heartbeat confirming listener is registered
```

## Root Cause Analysis

### Hypothesis

**Primary (most likely):** `MyAccessibilityService` is killed and restarted by Android (normal OS behavior for long-running services). On restart, `onServiceConnected()` fires again but `BlockingService` never re-registers its listener — the static listener reference in `MyAccessibilityService` is null after restart.

**Secondary:** `BlockingService` itself is killed by Android battery optimization (Doze mode, app standby buckets). Even with `START_STICKY`, the foreground notification may not be sufficient to survive aggressive OEM battery optimization (Samsung, Xiaomi, etc.).

**Tertiary:** Race condition — if both services restart around the same time, `BlockingService` may try to register before `MyAccessibilityService` is ready, failing silently.

### Suspected Files
| File | Reason |
|------|--------|
| `expo-accessibility-service/.../MyAccessibilityService.kt` | Static listener reference lost on service restart |
| `tied-siren-blocking-overlay/.../BlockingService.kt` | No re-registration logic on reconnect; possibly killed by Doze |
| `tied-siren-blocking-overlay/.../BlockingOverlayActivity.kt` | May need wake lock for reliable launch |
| `AndroidManifest.xml` (both modules) | Service flags, foreground service type, permissions |

---

## Proposed Fix

### Changes Required

**1. Listener re-registration on service reconnect**
- In `MyAccessibilityService.onServiceConnected()`: check if `BlockingService` is running and re-register its listener
- In `BlockingService`: implement `ServiceConnection` or broadcast-based re-registration when `MyAccessibilityService` restarts
- Replace static listener reference with a more robust pattern (e.g., `WeakReference` + periodic validation, or bind-based approach)

**2. ForegroundService hardening**
- Ensure `BlockingService` returns `START_STICKY` from `onStartCommand()`
- Add a partial `WakeLock` for critical detection windows
- Set `android:foregroundServiceType="specialUse"` (API 34+)
- Add `FOREGROUND_SERVICE_SPECIAL_USE` permission

**3. Watchdog mechanism**
- Add periodic self-check (every 5 min via `Handler.postDelayed`) that verifies listener is registered
- If listener is null, attempt re-registration automatically
- Log heartbeat to Sentry for gap detection

**4. Battery optimization exemption**
- Prompt user for `REQUEST_IGNORE_BATTERY_OPTIMIZATIONS` if not already granted
- Guide user to disable battery optimization for the app on OEM devices

**5. Lifecycle logging**
- Log every service lifecycle event: `onCreate`, `onDestroy`, `onStartCommand`, `onServiceConnected`, `onUnbind`
- Log listener registration/unregistration with timestamps
- Log each app detection -> blocked check -> overlay launch decision chain

### Risks
- `REQUEST_IGNORE_BATTERY_OPTIMIZATIONS` may trigger Google Play policy review if not justified
- WakeLock misuse can drain battery — must use partial lock with timeout
- OEM-specific battery optimization (Samsung, Xiaomi) may still kill services regardless of standard Android exemptions
